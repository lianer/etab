# eTab 卡片沙箱机制实现原理

## 概述

eTab项目的"卡片沙箱"是一个基于iframe的隔离环境，用于安全执行用户自定义的HTML/CSS/JavaScript代码。该机制在用户自定义UI的灵活性和主应用安全性之间取得平衡，是Web组件化架构中常见的安全沙箱模式。

## 核心架构

```
Card组件 → CardRenderer → IframeSandbox → iframe沙箱环境
```

- **Card组件**：负责卡片的拖拽、缩放和UI展示
- **CardRenderer**：根据卡片类型渲染对应的卡片内容
- **IframeSandbox**：实现iframe沙箱隔离和代码执行
- **iframe沙箱环境**：执行用户自定义代码的安全环境

## 1. 沙箱隔离机制

### 关键技术
HTML iframe + sandbox属性实现浏览器级隔离。

### 代码实现
```tsx
// IframeSandbox.tsx:99-105
<iframe
  ref={iframeRef}
  className={styles.iframe}
  srcDoc={srcDoc}
  sandbox="allow-scripts allow-same-origin"  // 关键安全属性
  title={`card-${card.id}`}
/>
```

### 安全属性配置
| 属性 | 允许 | 禁止 | 目的 |
|------|------|------|------|
| `allow-scripts` | JavaScript执行 | - | 支持用户代码运行 |
| `allow-same-origin` | 同源访问 | - | postMessage通信需要 |
| - | - | `allow-forms` | 防止表单提交 |
| - | - | `allow-popups` | 防止弹出窗口 |
| - | - | `allow-top-navigation` | 防止页面重定向 |

## 2. 动态内容构建

### 代码注入流程
```tsx
// IframeSandbox.tsx:16-69
const buildIframeContent = useCallback(() => {
  const { html = '', css = '', js = '' } = card.customCode || {};
  return `<!DOCTYPE html>
<html>
<head>
  <meta charset="UTF-8">
  <style>
    * { box-sizing: border-box; margin: 0; padding: 0; }
    body { font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif; }
    ${css}
  </style>
</head>
<body>
  ${html}
  <script>
    // 注入的Dashboard API和用户代码
    ${js}
  </script>
</body>
</html>`;
}, [card.customCode]);
```

### 特点
- **实时构建**：每次代码更新时动态生成完整HTML文档
- **零网络请求**：通过`srcDoc`属性直接注入内容
- **基础样式**：包含CSS重置和默认字体设置
- **代码分离**：HTML、CSS、JS分别注入到对应位置

## 3. API通信系统

### 沙箱内API（注入到iframe中）
```javascript
// IframeSandbox.tsx:32-49
const dashboardAPI = {
  // 发送消息到父页面
  postMessage(type, data) {
    window.parent.postMessage({ source: 'card', type, data }, '*');
  },

  // 异步获取应用状态
  getState() {
    return new Promise((resolve) => {
      const id = Math.random().toString(36);
      const handler = (e) => {
        if (e.data && e.data.type === 'state-response' && e.data.id === id) {
          window.removeEventListener('message', handler);
          resolve(e.data.state);
        }
      };
      window.addEventListener('message', handler);
      window.parent.postMessage({ source: 'card', type: 'get-state', id }, '*');
    });
  }
};
```

### 父页面监听器
```tsx
// IframeSandbox.tsx:77-87
useEffect(() => {
  const handler = (e: MessageEvent) => {
    if (e.data?.source !== 'card') return;  // 验证消息来源
    if (e.data?.type === 'error') {
      console.error(`Card error (${card.id}):`, e.data.data?.message);
      setHasError(true);  // 触发错误边界
    }
  };
  window.addEventListener('message', handler);
  return () => window.removeEventListener('message', handler);
}, [card.id]);
```

### 通信机制特点
1. **双向通信**：iframe ↔ 父页面的双向消息传递
2. **来源验证**：通过`source: 'card'`标识符防止伪造消息
3. **异步设计**：`getState()`返回Promise，支持异步状态获取
4. **唯一标识**：使用随机ID匹配请求响应

## 4. 安全与错误处理

### 多层防护体系

| 防护层 | 实现位置 | 机制 | 目的 |
|--------|----------|------|------|
| **执行超时** | IframeSandbox.tsx:9,53-59 | `SCRIPT_TIMEOUT_MS = 5000` | 防止无限循环或卡死 |
| **错误捕获** | IframeSandbox.tsx:61-65 | `try-catch`包裹用户代码 | 避免崩溃影响主应用 |
| **错误边界** | IframeSandbox.tsx:89-96 | `hasError`状态 + 重试按钮 | 用户友好的错误恢复 |
| **消息验证** | IframeSandbox.tsx:79 | `e.data?.source === 'card'` | 防止跨域消息干扰 |
| **沙箱限制** | IframeSandbox.tsx:103 | `sandbox`属性配置 | 限制危险操作 |

### 超时检测实现
```javascript
// 注入到iframe中的检测代码
const _startTime = Date.now();
const _checkTimeout = () => {
  if (Date.now() - _startTime > ${SCRIPT_TIMEOUT_MS}) {
    dashboardAPI.postMessage('error', { message: 'Script execution timeout' });
    return true;
  }
  return false;
};
```

### 错误边界UI
```tsx
// IframeSandbox.tsx:89-96
if (hasError) {
  return (
    <div className={styles.errorBoundary}>
      <p>卡片运行出错</p>
      <button className={styles.retryBtn} onClick={() => setHasError(false)}>重试</button>
    </div>
  );
}
```

## 5. 数据流与状态管理

### 代码编辑流程
1. **用户输入**：通过三标签编辑器（HTML/CSS/JS）输入代码
2. **临时存储**：代码保存到`localStorage`（键：`etab-card-code-${card.id}`）
3. **配置更新**：保存时调用`updateCardConfig()`更新卡片配置
4. **实时渲染**：配置更新触发IframeSandbox重新构建内容

### 渲染决策逻辑
```tsx
// CardRenderer.tsx:13-25
switch (card.cardType) {
  case 'clock':
    return <ClockCard cardId={card.id} config={card.cardConfig} />;
  case 'calendar':
    return <CalendarCard cardId={card.id} config={card.cardConfig} />;
  case 'todo':
    return <TodoCard cardId={card.id} config={card.cardConfig} />;
  case 'custom':
  case 'preset':
    return <IframeSandbox card={card} />;  // 自定义/预设代码入口
  default:
    return <div>未知卡片类型</div>;
}
```

### 数据结构

## 6. 设计优势与考量

### 优势
1. **真正的环境隔离**：iframe提供浏览器级安全边界，用户代码无法访问主应用DOM
2. **性能可控**：超时机制和错误边界防止脚本失控影响系统性能
3. **开发体验良好**：实时编辑、即时预览，支持代码高亮和分标签编辑
4. **扩展性强**：统一的API接口设计，支持未来添加更多功能
5. **数据持久化**：localStorage提供代码草稿保存，避免意外丢失

### 安全考量
1. **沙箱属性权衡**：`allow-same-origin`允许同源访问，方便通信但降低隔离性
2. **消息验证**：严格的来源验证防止恶意消息注入
3. **代码执行限制**：无`eval()`或`Function()`构造器使用，减少注入风险
4. **CSP考虑**：未来可考虑添加Content Security Policy进一步增强安全

### 限制与改进空间
1. **CSS隔离**：iframe样式隔离可能导致某些CSS继承问题
2. **性能开销**：移动端iframe创建和销毁有一定性能开销
3. **通信延迟**：postMessage通信相比直接调用有一定延迟
4. **功能限制**：沙箱限制某些高级API访问（如摄像头、地理位置等）

### 适用场景
1. **用户自定义UI**：允许用户创建个性化卡片
2. **教学演示**：实时代码编辑和预览
3. **插件系统**：安全执行第三方代码
4. **数据可视化**：自定义图表和展示组件

## 总结

eTab的卡片沙箱机制是一个成熟、安全的Web沙箱实现方案：

1. **核心技术**：iframe + sandbox属性提供基础隔离
2. **通信机制**：postMessage实现安全双向通信
3. **安全防护**：多层防护体系保障系统稳定性
4. **用户体验**：实时编辑、错误恢复、数据持久化

该设计在隔离性、功能性和开发体验之间取得了良好平衡，适合需要执行用户自定义代码的Web应用场景。随着Web标准发展，未来可考虑结合Web Components、Trusted Types等新技术进一步增强安全性和性能。
